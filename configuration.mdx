All settings live in `lumi_mint_bot_config.yaml`, created automatically next to the executable on first run (see [Installation & First Run](/installation)). Open it in any plain text editor.

<Tip>
  Edited the file while the bot is running? You don't need to restart. Choose **↺ Reload Config** from the main menu.
</Tip>

<Note title="Quoting rule">
  Every field marked `type: string` below must be wrapped in double quotes in the YAML file, for example `slug: "my-drop"`. Numbers (`float`, `integer`) and `true`/`false` values are written without quotes, for example `quantity_per_wallet: 1`.
</Note>

## Authentication

<ParamField path="authentication.license_key" type="string" required>
  Your Lumi license key. See [Getting a License Key](/getting-license-key). The bot won't start without a valid one.
</ParamField>

## API keys

<ParamField path="opensea_api_key" type="string" required>
  Your OpenSea API key, used for fetching drop and stage information, and also for pulling mint calldata at mint time. Required for OpenSea mode.

  <Steps>
    <Step title="Create or log into your OpenSea account">
      Go to [opensea.io](https://opensea.io) and connect a wallet to sign in.
    </Step>
    <Step title="Open the API key request page">
      Visit [docs.opensea.io/reference/api-keys](https://docs.opensea.io/reference/api-keys) and follow the link to request a key.
    </Step>
    <Step title="Fill out the request form">
      Provide your project details when asked. A short description like "personal minting tool" is fine.
    </Step>
    <Step title="Copy your key once issued">
      Paste it into the `opensea_api_key` field of your settings file.
    </Step>
  </Steps>
</ParamField>

<ParamField path="explorer_api_keys.ethereum" type="string">
  A free Etherscan API key ([get one here](https://etherscan.io/myapikey)), written as `ethereum: "YourApiKeyHere"`. Only used by **Custom Contract → Discover contract** mode, to fetch a verified contract's real ABI (with readable parameter names). If left blank, the bot falls back to guessing function signatures via 4byte.directory, it still works, just with less readable output.
</ParamField>

<ParamField path="discord_webhook_url" type="string">
  A Discord webhook URL, written as `discord_webhook_url: "https://discord.com/api/webhooks/..."`. If set, the bot posts a summary embed after each mint attempt. Leave as `""` to disable.
</ParamField>

## Wallets

<ParamField path="wallets" type="list of strings" required>
  Wallets available to mint from. See [Setting Up Your Wallets](/wallet-setup) for the full format (`label,0xprivatekey`).
</ParamField>

## Mint settings

These apply to **OpenSea**, **Blever**, and **Transient** modes.

<ParamField path="mint.slug" type="string" required>
  The collection's slug, the part of the drop's URL that identifies it. For example, for `opensea.io/collection/my-drop` the slug is `my-drop`, so you'd write `slug: "my-drop"`.
</ParamField>

<ParamField path="mint.chain" type="string" required default="ethereum">
  Which chain to mint on. Must exactly match one of the chain names you've listed under `rpcs` (e.g. `chain: "ethereum"`, `chain: "base"`, `chain: "polygon"`, `chain: "blast"`, `chain: "abstract"`).
</ParamField>

<ParamField path="mint.quantity_per_wallet" type="integer" required>
  How many NFTs each selected wallet should mint. Defaults to `1`.

  <Warning title="OpenSea's per-wallet limit is often cumulative, not per-stage">
    If your wallet is eligible for more than one OpenSea stage, for example both **GTD** (guaranteed) and **FCFS** (first come first served), check the "limit per wallet" on each stage carefully. That number is usually a running total across every stage you're eligible for, not a fresh allowance per stage.

    <Frame caption="GTD shows a limit of 1 per wallet, FCFS shows a limit of 2 per wallet. Minting 1 in GTD leaves only 1 remaining in FCFS, not 2.">
      <img src="/images/opensea-wallet-limit.png" />
    </Frame>

    The bot automatically checks what a wallet has already minted before attempting more, so you won't overshoot the real limit even if the stage's displayed number is misleading. See the [Minting Guide](/minting-guide#wait-for-eligibility-checks) for details on this check.
  </Warning>
</ParamField>

<ParamField path="mint.mint_price_eth" type="float" required>
  <strong>This is a price ceiling, not the price you'll pay.</strong> The bot compares this to the drop's real on-chain price and refuses to mint if the real price is higher.

  This is a price safeguard: if a project raises its mint price at the last minute, for example from a free mint to a paid one, the bot won't blindly pay whatever the contract now demands. It aborts instead, so you don't get charged more than you intended. Set this to the actual mint price (or higher, to be safe), or `0.0` for free mints. Defaults to `0.0`.
</ParamField>

## Custom contract settings

Only used by [Custom Contract mode](/custom-contract). See that page for the "Discover contract" flow, which can fill most of this in for you interactively instead.

<ParamField path="custom_contract.contract_address" type="string" required>
  The target contract's address, e.g. `contract_address: "0xYourContractAddress"`.
</ParamField>

<ParamField path="custom_contract.function" type="string">
  A function signature, e.g. `function: "mint(uint256)"`. If set, calldata is built from this plus `parameters`, and `hex` is ignored.
</ParamField>

<ParamField path="custom_contract.parameters" type="string">
  Semicolon-separated argument values matching `function`'s types, written as a single quoted string. Example for `mint(uint256)`, if the value is `3`: `parameters: "3"`. Example for `mint(address,uint256)`: `parameters: "0xABCD...;3"`. Required only if `function` is set.
</ParamField>

<ParamField path="custom_contract.hex" type="string">
  Full raw calldata (including the 4-byte function selector), e.g. `hex: "0xa0712d68..."`. Only used when `function` is left empty.
</ParamField>

<ParamField path="custom_contract.value_eth" type="float">
  ETH to send with the transaction. `0.0` for free mints. Defaults to `0.0`.
</ParamField>

<ParamField path="custom_contract.chain" type="string" default="ethereum">
  Which chain to mint on. Must match a key under `rpcs`.
</ParamField>

<ParamField path="custom_contract.start_time" type="integer (Unix timestamp)">
  When to count down to for Scheduled and Gas Monitor modes. This is a raw Unix timestamp (seconds since 1970), not a normal date. Use a converter like [unixtimestamp.com](https://www.unixtimestamp.com/) to convert a real date/time into this format. `0` or blank means "mint immediately, no countdown." Defaults to `0`.
</ParamField>

## Gas settings

<ParamField path="gas.mode" type="'rapid' or 'manual'" default="rapid">
  **rapid**: the bot reads the network's current base fee live and applies `multiplier` on top. Good default; adapts automatically as network conditions change.

  **manual**: the bot always uses the fixed `base_gwei` / `priority_gwei` values below, regardless of live network conditions.
</ParamField>

<ParamField path="gas.multiplier" type="float">
  Rapid mode only. Multiplies the live base fee. `1.0` = pay exactly the current base fee, `1.5` = pay 50% above it (safer during volatile gas spikes, but more expensive). Defaults to `1.2`.
</ParamField>

<ParamField path="gas.base_gwei" type="float">
  Manual mode: the maximum fee per gas unit, in gwei. Defaults to `1.0`.
</ParamField>

<ParamField path="gas.priority_gwei" type="float">
  Manual mode only: the "tip" paid to the block producer, in gwei. Defaults to `1.0`.

  <Tip>
    Most botters set `priority_gwei` equal to `base_gwei`. On an EIP-1559 chain, the actual tip you pay is capped at whatever room is left between your max fee (`base_gwei`) and the real network base fee, so setting your priority this high just means you're always offering the full amount you're willing to pay as a tip. That maximizes how competitively your transaction is ordered against other bots during a busy mint, up to the ceiling you already set.
  </Tip>

  <Tip>
    Not sure what numbers to plug in? Run `/gas-calculator` in the `🤖│bot-commands` channel in Discord to work out a gwei value.
  </Tip>
</ParamField>

<ParamField path="gas.gas_limit" type="integer">
  The computation budget for the transaction. Defaults to `0`.

  Set a fixed number (e.g. `160000`) to skip estimation and use exactly that.

  Set to `0` for **"snipe mode"**: the bot repeatedly simulates the transaction until the contract actually accepts it. This is intentional and useful when you don't know the exact second a mint opens, but expect rapid repeated log output, and be aware it uses real RPC calls (can hit rate limits on free public endpoints).
</ParamField>

<ParamField path="gas.flashbots_protect" type="boolean">
  Routes the transaction privately via Flashbots Protect. If it would revert, you pay no gas. **Ethereum mainnet only, has no effect on any other chain.** Does not apply to MEV Tip mode, which uses its own builder list instead. Defaults to `false`.

  **A. Normal mode.** Your transaction goes to the public mempool and competes on gas price. If it fails (reverts), you still pay a partial gas fee for the attempt. Each wallet submits individually.

  **B. Flashbots Protect.** Your transaction goes through Flashbots' public RPC instead of the normal mempool (slightly slower response time) and still competes on gas price. If it would fail, it's simply never included in a block, so you pay zero gas for it. Each wallet still submits individually.

  **Example**, both configured with `base_gwei: 80`:
  - **A.** Mint succeeds: wallet pays gas at 80 gwei. Mint fails: wallet still pays a partial gas fee for the failed attempt.
  - **B.** Mint succeeds: wallet pays gas at 80 gwei. Mint fails: wallet pays 0 gas.
</ParamField>

## Per-wallet overrides

Use this when one wallet needs different settings than the rest, for example a wallet with a smaller allowlist allocation that should mint fewer units, or a wallet you want to prioritize with a higher gas tip to make sure it lands first.

<ParamField path="wallet_overrides.<wallet_label>.base_gwei" type="float">
  Override `gas.base_gwei` for one specific wallet (by its label).
</ParamField>
<ParamField path="wallet_overrides.<wallet_label>.priority_gwei" type="float">
  Override `gas.priority_gwei` for one specific wallet.
</ParamField>
<ParamField path="wallet_overrides.<wallet_label>.quantity_per_wallet" type="integer">
  Override `mint.quantity_per_wallet` for one specific wallet.
</ParamField>

Any wallet not listed here just uses the global settings above. Example:

```yaml
wallet_overrides:
  my_wallet:
    base_gwei: 5.0
    priority_gwei: 2.0
    quantity_per_wallet: 3
```

If you don't need any per-wallet overrides, comment the whole block out and every wallet just uses the global settings:

```yaml
wallet_overrides:
  # my_wallet:
  #   base_gwei: 5.0
  #   priority_gwei: 2.0
  #   quantity_per_wallet: 3
```

## Bundle / MEV settings

Used by [MEV Tip mode](/mint-modes#e-mev-tip).

<ParamField path="bundle.tip_eth" type="float" required>
  ETH tip paid to the block builder per bundle submission. Size this against what competitor bots are likely tipping for the same drop. The default (`0.001`) is intentionally conservative and may lose every block during a competitive mint.
</ParamField>

<ParamField path="bundle.tip_wallet" type="string">
  Optionally, a dedicated wallet to pay the tip from instead of one of your mint wallets, written as `tip_wallet: "label"` or `tip_wallet: "label,0xprivatekey"`. Leave `""` to use the first eligible mint wallet.
</ParamField>

<ParamField path="bundle.gas_limit" type="integer">
  Manual gas limit for the bundle's mint transactions. `0` = auto-estimate. Defaults to `0`.
</ParamField>

<ParamField path="bundle.max_blocks" type="integer" required>
  How many blocks in a row to keep resubmitting the bundle if it doesn't land. Defaults to `5`.
</ParamField>

## RPC endpoints

<ParamField path="rpcs.<chain_name>" type="list of URLs" required>
  RPC endpoints for each chain you mint on. The bot ships with working public endpoints for `ethereum` and `base` already filled in:

  ```yaml
  rpcs:
    ethereum:
      - "https://ethereum-rpc.publicnode.com"
      - "https://eth.drpc.org"
    base:
      - "https://base-rpc.publicnode.com"
      - "https://base.drpc.org"
  ```

  To add a new chain, add a new key with the same name OpenSea uses for it (`polygon`, `blast`, `abstract`, etc.) and at least one RPC URL.

  <Tip>
    Listing multiple URLs per chain isn't just a backup. Wallets are assigned RPCs round-robin, so multiple URLs spread request load across endpoints and reduce the chance of hitting a rate limit during a busy mint.
  </Tip>
</ParamField>

## Proxy

<ParamField path="proxy" type="list of strings">
  A list of proxies in `login:password@hostname:port` format. The bot distributes them across wallets in round-robin order. Leave the list empty (or comment all entries out) to disable proxies entirely.

  ```yaml
  proxy:
    - "user1:pass1@proxy.example.com:8080"
    - "user2:pass2@proxy.example.com:8081"
  ```

  To disable, comment all entries out:

  ```yaml
  proxy:
    # - "login:password@hostname:port"
  ```

  <Warning>
    Proxy settings only apply to OpenSea sign-in and eligibility checks. **The actual mint transaction is always sent directly, unproxied**, for lower latency. Don't rely on this setting for full traffic anonymization.
  </Warning>
</ParamField>
